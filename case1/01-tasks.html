개발 파트별 Task 정의

RFP 기반 프로젝트의 PM/FE/BE/DB/Infra 각 파트별 작업 정의서로, 요구사항을 실제 수행 단위의 Task와 산출물 중심으로 분해할 때 활용한다.

1. 작성 목적

RFP 문서는 범위와 기대 결과를 제시하지만, 실제 프로젝트 운영을 위해서는 파트별 작업 단위와 산출물을 명확히 정의해야 한다. 아래 표는 PM, FE, BE, DB, Infra 각 파트가 착수 이후 어떤 일을 어떤 결과물로 남겨야 하는지 정리한 기준안이다.

실무에서는 이 표를 그대로 사용하는 것보다, 사업 특성에 맞춰 Task 명칭과 산출물 수준을 조정하는 것이 좋다. 예를 들어 공공사업은 산출물 중심, 민간 SaaS는 반복 개발과 배포 체계 중심으로 세분화한다.

2. PM Task

Task ID 작업명 상세 내용 산출물
PM PM-001 프로젝트 착수 Kick-off 준비, 이해관계자 식별, 의사결정 체계와 커뮤니케이션 채널을 확정한다. 착수보고서, 이해관계자 목록, 회의체 운영안
PM PM-002 RFP 분석 및 요구사항 정의 RFP 범위, 제약사항, 우선순위를 구조화하고 기능/비기능 요구사항을 정리한다. 요구사항 정의서, 범위 목록, 이슈 로그
PM PM-003 프로젝트 계획 수립 WBS, 일정, 투입 인력, 의존성, 마일스톤을 포함한 실행 계획을 수립한다. 프로젝트 계획서, WBS, 일정표
PM PM-004 형상관리 계획 브랜치 전략, 배포 승인 절차, 문서/소스 형상 기준과 변경관리 절차를 정의한다. 형상관리 계획서, 브랜치 정책, 변경관리 프로세스
PM PM-005 품질관리 계획 검토/테스트/승인 절차와 품질 지표, 산출물 점검 기준을 수립한다. 품질관리 계획서, 검수 체크리스트, 품질 지표표
PM PM-006 리스크 관리 일정, 범위, 인력, 기술 리스크를 사전 식별하고 대응 전략 및 보고체계를 운영한다. 리스크 등록부, 대응계획서, 주간 리스크 리포트

3. FE Task

Task ID 작업명 상세 내용 산출물
FE FE-001 UI/UX 요구사항 분석 화면 목록, 사용자 흐름, 접근성, 반응형 요구사항을 식별하고 화면 우선순위를 정한다. 화면 정의서, 사용자 플로우, UI 요구사항 목록
FE FE-002 FE 아키텍처 설계 라우팅, 상태관리, API 연동 구조, 폴더 정책, 공통 컴포넌트 전략을 정의한다. FE 아키텍처 문서, 폴더 구조안, 상태관리 전략서
FE FE-003 디자인 시스템 구축 공통 UI 컴포넌트, 토큰, 스타일 가이드, 접근성 규칙을 정비한다. 디자인 시스템 가이드, 공통 컴포넌트 목록, UI 규칙서
FE FE-004 화면 개발 요구 화면을 개발하고 API 연동, 예외 처리, 입력 검증 로직을 구현한다. 화면 소스코드, 화면별 구현 체크리스트
FE FE-005 FE 테스트 단위 테스트, 화면 시나리오 테스트, 브라우저 호환성 및 접근성 검증을 수행한다. 테스트 결과서, 결함 목록, 개선 이력
FE FE-006 성능 최적화 번들 크기, 렌더링 병목, 캐싱, 이미지 최적화, Core Web Vitals 개선을 수행한다. 성능 개선 보고서, 최적화 체크리스트, 측정 결과
Example

RFP에 “대민 서비스 모바일 대응”이 있으면 FE Task에는 반응형 브레이크포인트 정의, 모바일 입력 UX 개선, 디바이스 테스트 항목이 추가되어야 한다.

4. BE Task

Task ID 작업명 상세 내용 산출물
BE BE-001 비즈니스 로직 분석 업무 규칙, 승인 절차, 예외 케이스, 배치/연계 요구를 구조화한다. 업무 규칙 정의서, 서비스 목록, 연계 요구사항 표
BE BE-002 API 설계 도메인별 API 목록, 요청/응답 모델, 오류 코드, 인증 방식을 정의한다. API 명세서, OpenAPI 문서, 오류 코드 표
BE BE-003 BE 아키텍처 설계 레이어 구조, 트랜잭션 정책, 모듈 분리, 메시징/배치 구조를 설계한다. BE 아키텍처 설계서, 모듈 구성도, 기술 결정 기록
BE BE-004 비즈니스 로직 구현 서비스, 도메인 로직, 배치, 외부 연계 처리와 예외 처리 로직을 구현한다. 서버 소스코드, 구현 체크리스트, 코드 리뷰 이력
BE BE-005 API 통합 테스트 시스템 간 호출, 장애 상황, 데이터 정합성, 응답시간 기준을 검증한다. 통합 테스트 결과서, 인터페이스 검증표
BE BE-006 보안 설계 인증/인가, 암호화, 감사로그, 시크릿 관리, 취약점 대응 기준을 설계한다. 보안 설계서, 권한 정책서, 보안 점검 항목

5. DB Task

Task ID 작업명 상세 내용 산출물
DB DB-001 데이터 요구사항 분석 업무 데이터, 보존주기, 개인정보, 연계 데이터, 정합성 요구를 식별한다. 데이터 요구사항 정의서, 데이터 항목 목록
DB DB-002 개념적 데이터 모델링 핵심 엔티티, 주요 관계, 업무 단위 기준의 데이터 구조를 추상화한다. 개념 ERD, 엔티티 정의서
DB DB-003 논리적 모델링 속성, 식별자, 관계 차수, 정규화 수준을 반영해 논리 스키마를 설계한다. 논리 ERD, 속성 정의서
DB DB-004 물리적 모델링 테이블, 컬럼 타입, PK/FK, 인덱스, 파티셔닝, 운영 정책을 구체화한다. 물리 모델, 테이블 정의서, DDL 초안
DB DB-005 DB 성능 튜닝 쿼리 계획, 인덱스 효율, 통계 관리, 배치 성능, 용량 증가 대응을 점검한다. 튜닝 보고서, 인덱스 설계서, 성능 측정 결과

6. Infra Task

Task ID 작업명 상세 내용 산출물
Infra IF-001 인프라 요구사항 분석 가용성, 성능, 보안, 운영, 백업/복구, 배포 요구사항을 정리한다. 인프라 요구사항 정의서, 용량 산정서
Infra IF-002 시스템 아키텍처 설계 네트워크 구간, 서버 역할, 이중화, 외부 연계, 운영 환경 구성을 설계한다. 시스템 구성도, 인프라 아키텍처 문서
Infra IF-003 CI/CD 구축 빌드, 테스트, 배포 자동화, 승인 절차, 환경별 배포 전략을 구성한다. 파이프라인 정의서, 배포 스크립트, 운영 절차서
Infra IF-004 모니터링/로깅 시스템/애플리케이션/보안 로그 수집과 경보 기준, 대시보드를 구축한다. 모니터링 대시보드, 로그 정책서, 알림 기준표
Infra IF-005 보안 인프라 방화벽, WAF, VPN, 접근통제, 인증서, 취약점 점검 체계를 구축한다. 보안 구성도, 접근통제 정책서, 취약점 조치 내역
RFP에 망분리, DRM, 개인정보 암호화, DR센터 구축 요구가 있으면 Infra Task는 별도 하위 작업으로 분해하는 것이 안전하다.

7. 활용 예시 및 작성 팁

Example Output

공통 요구사항 “외부기관 연계 API 12종 구축”이 포함되면 PM은 연계 일정 조정, BE는 API 명세/보안 설계, DB는 연계 이력 테이블 설계, Infra는 망연계 구간 검토까지 함께 계획해야 한다.

작업 정의서는 단순 목록보다 “누가, 무엇을, 어떤 문서/코드로 남기는가”가 보여야 실행력이 높아진다. 따라서 상세 내용과 산출물을 함께 적는 형식을 유지하는 것이 중요하다.

8. AI 프롬프트 — 파트별 Task 도출

Prompt Template
다음은 RFP 기반 프로젝트의 요구사항이다.

[입력 정보]
- 프로젝트 개요:
- 주요 기능:
- 비기능 요구사항:
- 연계 시스템:
- 보안/인프라 제약:

위 정보를 바탕으로 PM, FE, BE, DB, Infra 파트별 Task를 도출해줘.
조건은 다음과 같다.
1) 각 Task는 Task ID, 작업명, 상세 내용, 산출물 형식으로 작성
2) PM/FE/BE는 각 6개, DB/Infra는 각 5개 제시
3) RFP 문장에서 직접 유도된 작업이 보이도록 설명
4) 산출물은 문서/코드/설정/테스트 결과 중심으로 구체화
5) 결과는 HTML 테이블에 바로 옮길 수 있게 표 형태로 정리

요구사항 원문을 함께 넣으면 AI가 파트별 누락 작업을 줄이고, 산출물 중심으로 더 정확하게 분해할 수 있다.
