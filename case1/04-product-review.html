제품/플랫폼 검토

기술 스택 비교 분석과 선정 기준을 정리하여, 제안 단계 또는 착수 초기의 기술 의사결정을 구조적으로 설명하기 위한 검토 문서다.

1. 기술 스택 검토 매트릭스

기술 선정은 단순 선호가 아니라 인력 역량, 라이선스, 운영성, 확장성, 생태계, 요구사항 적합성을 함께 고려해야 한다. 아래 표는 RFP 검토 문서에서 자주 사용하는 대표 기술 비교 예시다.

구분 기술 장점 단점 라이선스 권장
FEReact생태계가 크고 컴포넌트 재사용성과 인력 수급이 좋다.구조 표준을 별도로 정하지 않으면 프로젝트별 편차가 크다.MIT○
FEVue학습 곡선이 낮고 빠른 화면 개발에 유리하다.대규모 구조화와 기업형 레퍼런스는 React 대비 상대적으로 적다.MIT△
FENext.jsSSR/SSG, 라우팅, 성능 최적화 기능이 내장되어 있다.런타임과 배포 전략 이해가 필요하고 단순 SPA에는 과할 수 있다.MIT○
BESpring Boot엔터프라이즈 기능, 안정성, 보안/배치/연계 레퍼런스가 풍부하다.초기 설정과 운영 리소스가 비교적 무겁다.Apache 2.0○
BENestJSTypeScript 기반 구조화가 좋고 FE와 언어 통일이 가능하다.초대형 트랜잭션/배치 환경 레퍼런스는 Spring보다 적다.MIT△
BEFastAPI개발 생산성과 문서 자동화가 뛰어나고 ML 연계에 유리하다.대규모 조직 표준화와 장기 운영 체계는 추가 보완이 필요하다.MIT△
DBPostgreSQL표준 준수와 확장성이 높고 JSON/분석 기능도 우수하다.운영팀 경험이 부족하면 튜닝 난도가 있을 수 있다.PostgreSQL License○
DBMySQL운영 인력이 많고 보편적이며 구축 경험을 확보하기 쉽다.복잡한 분석성 쿼리나 고급 확장 기능은 제한적일 수 있다.GPL / Commercial△
DBMongoDB유연한 스키마와 문서 저장 구조로 빠른 변경 대응이 쉽다.정형 데이터와 강한 트랜잭션 요구에는 부적합할 수 있다.SSPL△
CacheRedis성능이 높고 자료구조가 다양해 세션/캐시/큐에 폭넓게 활용 가능하다.메모리 중심 구조라 용량 계획이 중요하다.RSAL 계열 / OSS 변형 검토 필요○
CacheMemcached구조가 단순하고 캐시 전용으로 가볍다.기능이 제한적이라 확장 활용성이 낮다.BSD△
ContainerKubernetes확장성, 표준성, 멀티서비스 운영에 강하다.운영 복잡도가 높고 학습 비용이 크다.Apache 2.0○
ContainerECSAWS 환경에서 비교적 단순하게 컨테이너 운영이 가능하다.클라우드 종속성이 커지고 세밀한 제어는 제한될 수 있다.AWS Managed△
CI/CDGitHub Actions저장소 연계가 쉽고 선언형 워크플로 구성에 유리하다.복잡한 사내망 환경이나 레거시 통합은 Jenkins보다 제약이 있다.GitHub Service○
CI/CDJenkins온프레미스/폐쇄망/복잡한 파이프라인 커스터마이징에 강하다.운영 관리 부담과 플러그인 유지보수 비용이 크다.MIT△
MonitoringDatadogAPM, 로그, 메트릭 통합이 편하고 구축 속도가 빠르다.장기적으로 비용이 높아질 수 있다.Commercial△
MonitoringPrometheus + Grafana오픈소스 기반으로 유연하고 비용 통제가 용이하다.초기 구축과 운영 표준화가 필요하다.Apache 2.0 / AGPL○

2. 선정 기준

주요 선정 기준
  • RFP 요구 적합성: 성능, 보안, 운영, 연계 요건 만족 여부
  • 조직 역량: 내부 개발자/운영자 숙련도와 외부 인력 수급성
  • 비용 구조: 라이선스, 운영비, 교육비, 전환 비용
  • 유지보수성: 장기 운영 시 표준화와 기술 부채 관리 가능성
  • 확장성: 서비스 확대, 기능 추가, 트래픽 증가 대응 가능성
Example Selection Logic

대규모 공공업무 시스템처럼 복잡한 권한, 감사로그, 배치, 외부기관 연계가 많은 경우에는 Spring Boot + PostgreSQL + Redis + Kubernetes + Prometheus/Grafana 조합이 자주 권장된다.

라이선스는 기술 우수성만큼 중요하다. 특히 SSPL, 상용 SaaS, 클라우드 종속형 서비스는 발주기관 정책과 예산 구조에 맞는지 반드시 별도 검토해야 한다.

- BE: Spring Boot, NestJS, FastAPI

1) 각 항목별 장점, 단점, 라이선스, 권장 여부를 표로 작성
