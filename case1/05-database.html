DB 설계

데이터 모델링과 DB 설계를 개념적·논리적·물리적 수준으로 정리하여, 요구사항을 안정적인 데이터 구조로 전환하기 위한 기준 문서다.

1. 데이터 모델링 프로세스

데이터 모델링은 업무 개념을 저장 구조로 옮기는 과정이다. RFP 기반 프로젝트에서는 화면과 API보다 먼저 데이터 기준이 흔들리지 않아야 전체 설계가 안정된다.

단계 목적 주요 산출물
개념적 모델링업무의 핵심 엔티티와 관계를 추상화한다.개념 ERD, 엔티티 목록
논리적 모델링속성, 식별자, 관계 차수, 정규화 수준을 정의한다.논리 ERD, 속성 정의서
물리적 모델링실제 DBMS 기준의 테이블/컬럼/인덱스/제약조건을 설계한다.테이블 정의서, 인덱스 설계서, DDL
업무 담당자 인터뷰와 화면/API 정의서를 함께 보면 엔티티 누락을 줄일 수 있다. 특히 이력, 첨부파일, 권한, 로그성 데이터는 초기 모델링에서 빠지기 쉽다.

2. 엔티티 도출 예시

아래 예시는 RFP 기반의 “민원/신청 서비스”를 가정한 가상 엔티티 목록이다.

  • User: 사용자 기본 정보, 인증 식별자, 상태
  • Organization: 소속 기관/부서 정보
  • Application: 신청/접수 기본 정보
  • ApplicationItem: 신청 상세 항목
  • Approval: 승인 단계, 승인자, 결과
  • Attachment: 업로드 파일 정보
  • AuditLog: 주요 행위 이력 및 변경 로그
Example

RFP에 “결재선 관리”와 “첨부문서 제출”이 있다면 Approval, Attachment 엔티티는 별도 핵심 엔티티로 분리하는 편이 일반적이다.

3. 논리적 ERD 설명

Mermaid erDiagram
erDiagram
    ORGANIZATION ||--o{ USER : has
    USER ||--o{ APPLICATION : creates
    APPLICATION ||--o{ APPLICATION_ITEM : contains
    APPLICATION ||--o{ APPROVAL : has
    APPLICATION ||--o{ ATTACHMENT : attaches
    USER ||--o{ AUDIT_LOG : triggers

    ORGANIZATION {
        bigint org_id PK
        varchar org_name
        varchar org_type
    }
    USER {
        bigint user_id PK
        bigint org_id FK
        varchar login_id
        varchar user_name
        varchar status
    }
    APPLICATION {
        bigint application_id PK
        bigint user_id FK
        varchar application_no
        varchar status
        datetime submitted_at
    }

논리 ERD에서는 관계의 의미와 업무 규칙이 드러나야 한다. 예를 들어 하나의 신청은 여러 승인 단계와 첨부파일을 가질 수 있고, 사용자 행위는 감사 로그로 추적 가능해야 한다.

테이블명 설명 주요 컬럼 비고

5. 인덱스 설계

  • tb_user(login_id): 로그인/사용자 조회 빈도가 높으므로 Unique Index 적용
  • tb_application(application_no): 신청번호 기반 단건 조회를 위해 Unique Index 적용
  • tb_application(user_id, status): 사용자별 진행 상태 조회 최적화
  • tb_approval(application_id, result): 승인 상태별 목록 조회 최적화
  • tb_audit_log(created_at): 기간별 로그 검색과 보관 정책 대응
인덱스는 많을수록 좋은 것이 아니다. 조회 성능과 쓰기 비용의 균형을 고려해, 실제 조회 패턴 기준으로 설계해야 한다.

6. DDL 예시

SQL Example
CREATE TABLE tb_application (
    application_id BIGSERIAL PRIMARY KEY,
    user_id BIGINT NOT NULL,
    application_no VARCHAR(30) NOT NULL,
    status VARCHAR(20) NOT NULL,
    submitted_at TIMESTAMP NULL,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT uq_application_no UNIQUE (application_no),
    CONSTRAINT fk_application_user FOREIGN KEY (user_id)
        REFERENCES tb_user(user_id)
);

CREATE INDEX idx_application_user_status
    ON tb_application (user_id, status);

CREATE TABLE tb_approval (
    approval_id BIGSERIAL PRIMARY KEY,
    application_id BIGINT NOT NULL,
    approver_id BIGINT NOT NULL,
    result VARCHAR(20) NOT NULL,
    approved_at TIMESTAMP NULL,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT fk_approval_application FOREIGN KEY (application_id)
        REFERENCES tb_application(application_id)
);
