Front-End 설계

FE 아키텍처, 컴포넌트 구조, 상태관리, 화면 정의를 기준으로 RFP 요구사항을 실제 구현 가능한 수준까지 구체화하기 위한 설계 문서입니다.

1. FE 아키텍처 개요

본 프로젝트의 Front-End는 React + Next.js 기반으로 구성하며, 서버 사이드 렌더링과 라우팅 표준화를 통해 초기 로딩 성능과 유지보수성을 동시에 확보합니다. 전역 상태는 Zustand, 서버 상태는 TanStack Query로 분리하여 상태 책임을 명확히 하고, UI 스타일은 Tailwind CSS를 적용해 일관된 디자인 토큰과 생산성을 확보합니다.

컴포넌트 구조는 Atomic Design을 채택하여 공통 UI 재사용성을 높이고, 화면 단위 요구사항이 증가해도 구조적으로 확장 가능한 FE 아키텍처를 지향합니다.

Framework: React + Next.js
State: Zustand + TanStack Query
Style: Tailwind CSS
Component: Atomic Design

4. 컴포넌트 계층 구조

Atomic Design은 작은 단위의 UI부터 화면 템플릿까지 계층적으로 구성하는 방법론입니다. 이를 적용하면 컴포넌트 책임이 분명해지고, 기능 추가 시 기존 UI를 재조합하는 방식으로 확장하기 쉬워집니다.

계층별 정의

  • Atoms: Button, Input, Badge, Icon 같은 최소 단위 요소
  • Molecules: SearchBox, FormField, UserCardHeader처럼 atoms 조합 단위
  • Organisms: Header, Sidebar, UserTable, BoardDetailPanel 같은 화면 구성 블록
  • Templates: 페이지 레이아웃, 섹션 배치, 공통 슬롯 구조
  • Pages: 실제 비즈니스 데이터와 라우팅이 결합된 화면 단위

예를 들어 게시판 상세 화면은 atoms(Button, Badge), molecules(AuthorInfo, CommentForm), organisms(BoardContent, CommentList), template(BoardDetailTemplate), page(boards/[id]) 순으로 조합됩니다.

6. 상태관리 전략 설명

상태는 성격별로 분리합니다. 화면 전역에서 즉시 반응해야 하는 UI 상태(사이드바 열림, 필터 조건, 선택 상태)는 Zustand에서 관리하고, 서버 API에서 조회되는 데이터는 TanStack Query에서 캐싱 및 재조회 정책과 함께 관리합니다. 이 분리는 불필요한 전역 상태 비대화를 막고, API 변경 시 영향 범위를 줄이는 데 유리합니다.

또한 인증 정보, 사용자 세션, 공통 코드 테이블은 별도 store와 query key 체계로 관리해 화면 간 데이터 일관성을 보장합니다. 등록/수정/삭제 이후에는 invalidateQueries 정책을 적용해 최신 데이터를 자동 반영하도록 설계합니다.
