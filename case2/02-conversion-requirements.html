전환 요구사항 및 TO-BE 정의

전환 유형과 범위를 명확히 정의하고, AS-IS와 TO-BE의 차이를 비교하여 실질적인 전환 전략과 리스크를 도출합니다.

1. 전환 유형 정의

전환 유형 설명 적용 상황 특징
Lift & Shift 기능 변경을 최소화하고 기존 시스템을 인프라 중심으로 이전 일정이 짧고 현행 구조 유지가 필요한 경우 빠르지만 구조적 개선 폭은 작음
Re-platform 애플리케이션은 유지하되 미들웨어/플랫폼을 교체 운영 환경 노후화, 클라우드 전환 필요 시 운영 효율은 개선되나 일부 호환성 검증 필요
Re-architect 핵심 구조를 서비스 지향 또는 현대적 아키텍처로 재설계 확장성, 성능, 유지보수성 개선이 핵심 목표인 경우 효과가 크지만 분석과 검증 비용이 높음
Rebuild 업무 요구는 유지하고 시스템을 사실상 재구축 기술 부채가 심각하거나 재사용 가치가 낮은 경우 자유도는 높지만 범위 통제가 중요

2. 전환 범위 정의

구분 항목 설명
PM 전환 유형 Lift & Shift, Re-platform, Re-architect, Rebuild 중 프로젝트 기본 방향을 정의한다.
공통 전환 범위 시스템, 기능, 데이터, 인프라, 외부 연동, 운영 체계 중 실제 전환 대상 범위를 명시한다.
공통 전환 기준 기능 동등성, 성능 목표, 운영 연속성, 보안 수준 등 완료 판정 기준을 정의한다.
공통 전환 제외 폐기 기능, 타 시스템 이관 대상, 별도 사업 예정 범위를 명확히 분리한다.
PM 전환 제약 예산, 일정, 라이선스, 인력, 병행 운영 기간, 다운타임 허용 시간 같은 제약을 정리한다.

3. AS-IS vs TO-BE 비교 매트릭스

영역 AS-IS TO-BE 전환 전략
FE 서버사이드 렌더링 중심, 화면별 공통 규칙 부족 컴포넌트 기반 SPA 또는 하이브리드 구조 핵심 화면부터 단계 전환, 디자인 토큰 우선 정립
BE 모놀리식 구조, 업무 로직 결합도 높음 계층 분리 강화, 모듈화된 서비스 구조 도메인 경계 기준으로 점진 분리
DB 벤더 종속 SQL/프로시저 다수 표준 SQL 중심, 이관 자동화 체계 확보 스키마 변환 + 데이터 검증 스크립트 병행
Infra 온프레미스 수동 배포 클라우드 또는 자동화된 표준 운영 환경 이중 환경 운영 후 점진 전환
공통 인증 레거시 세션 기반, 통합 정책 미흡 표준 인증 체계와 접근 통제 강화 기존 계정 체계와 호환 가능한 통합 인증 설계
Infra 모니터링 장애 후 수동 확인 중심 로그/메트릭/알람 통합 관제 운영 초기부터 대시보드와 알람 기준 수립
Infra CI/CD 배포 스크립트 및 수작업 절차 혼재 형상-빌드-배포 자동화 파이프라인 브랜치 전략과 환경별 승인 절차 정의
공통 보안 보안 점검이 배포 후속 활동 개발 단계 내 보안 내재화 SAST/DAST/권한 정책을 초기 설계에 반영

4. 전환 리스크 평가

전환 리스크는 기술 문제보다 누락된 범위, 숨은 업무 규칙, 데이터 품질, 병행 운영 절차에서 더 자주 발생한다. 특히 AS-IS 문서가 오래되었거나 핵심 담당자가 제한된 경우, 전환 범위는 단계적으로 확정해야 하며, 고난도 영역은 PoC 또는 파일럿 전환으로 먼저 검증하는 것이 안전하다.
  • 프로시저/배치에 숨어 있는 업무 규칙 누락
  • 외부 연동 전문 규격 변경 부담
  • 운영 배포 절차와 새 CI/CD 간 충돌
  • 성능 목표 미정의로 인한 재개발 확대
  • 전환 제외 범위가 불분명해 생기는 추가 요구
