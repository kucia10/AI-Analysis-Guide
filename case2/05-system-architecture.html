시스템 구성안 (TO-BE)

AS-IS 구조에서 TO-BE 구조로 어떻게 전환할지 계층별 변화 포인트와 신규 구성 원칙을 설명합니다.

1. 전환 아키텍처 개요

TO-BE 아키텍처는 단순한 기술 교체가 아니라 운영 자동화, 장애 대응, 보안 통제를 포함하는 구조 변화여야 한다. 기존 시스템의 업무 동등성을 유지하면서도 FE, BE, DB, Infra 계층을 독립적으로 개선할 수 있는 형태가 적합하다.

2. AS-IS → TO-BE 계층별 비교

계층 AS-IS TO-BE 전환 포인트
Web Server단일 웹 서버 또는 수동 확장표준화된 웹 게이트웨이 및 이중화정적 자산, 라우팅, SSL 종료 구조 정리
App Server모놀리식 WAS 중심모듈형 또는 컨테이너 기반 실행 환경배포 단위 축소와 무중단 배포 지원
FE페이지 단위 템플릿 구조컴포넌트 기반 FE 아키텍처공통 UI 규칙과 API 연동 계층 분리
BE업무 로직 결합도 높음도메인 중심 서비스 계층 분리테스트 용이성과 API 표준화 확보
DB벤더 종속 오브젝트 다수표준 SQL 및 이관 자동화 중심프로시저 축소, 데이터 검증 체계 강화
Cache부재 또는 로컬 캐시 위주공통 캐시 계층 도입조회 성능 개선 및 세션 분리
Monitoring부분 로그 수집로그/메트릭/트레이싱 통합 관제장애 탐지와 원인 분석 시간 단축
CI/CD수작업 빌드/배포자동화 파이프라인과 승인 절차반복 배포 품질과 추적성 확보

3. TO-BE Mermaid 아키텍처 다이어그램

Example
flowchart LR
    U[사용자] --> WAF[WAF / LB]
    WAF --> FE[FE Web/App]
    FE --> API[API Gateway]
    API --> APP1[BE Service A]
    API --> APP2[BE Service B]
    APP1 --> CACHE[Redis Cache]
    APP1 --> DB[(Primary DB)]
    APP2 --> DB
    DB --> REP[(Read Replica)]
    APP1 --> MQ[Batch / Message Queue]
    APP2 --> EXT[외부 연계 시스템]
    APP1 --> MON[Monitoring / Logging]
    APP2 --> MON

4. 인프라 리소스 사이징

구성 요소 수량 권장 사양 비고
웹/프론트 서버22vCPU / 4GB이중화 및 무중단 배포 고려
애플리케이션 서버2~44vCPU / 8~16GB업무 피크에 따라 오토스케일 가능
DB Primary18vCPU / 32GB / SSD쓰기 성능과 백업 정책 우선
DB Replica18vCPU / 32GB / SSD조회 분리 및 장애 대응
Cache1~22vCPU / 4GB세션/조회 캐시 분리 가능
관제 스택14vCPU / 8GB로그 적재량에 따라 확장

5. 데이터 흐름도 설명

사용자 요청은 웹 계층을 거쳐 API 게이트웨이로 진입하고, 이후 도메인별 BE 서비스에서 처리된다. 읽기 집중 구간은 캐시와 Replica를 우선 활용하며, 외부 연계는 비동기 큐 또는 표준 API를 통해 격리한다. 이 구조는 전환 초기에도 기존 시스템과 병행 운영하기 쉽도록 인터페이스 분리를 전제로 한다.

6. 보안 전환 설계

보안 설계는 인증 통합, 네트워크 구간 분리, 비밀정보 관리, 감사 로그 확보를 포함해야 한다. 특히 관리자 화면, 배치 계정, 외부 연동 계정은 기존 방식 그대로 이관하지 말고 권한 체계와 계정 수명주기를 다시 정의한다.

7. 롤백 아키텍처

컷오버 직후 즉시 복귀가 가능하도록 현행 시스템을 일정 기간 유지하고, 데이터 이관 시점별 스냅샷/백업을 확보한다. 라우팅 스위치, DB 백업, 배치 중단/재개 절차를 사전에 정의하면 장애 시 신규 구조를 전면 중단하지 않고 빠르게 복구할 수 있다.
