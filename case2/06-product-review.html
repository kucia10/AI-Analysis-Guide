제품/플랫폼 검토 (컨버팅 관점)

최신 기술 도입보다 실제 전환 가능성과 이식성, 운영 안정성을 우선하여 제품과 플랫폼을 검토하는 기준 문서입니다.

1. 컨버팅 관점 검토 원칙

컨버팅 프로젝트의 기술 검토는 최신성보다 전환 가능성, 호환성, 운영 지속성을 우선해야 한다. 가장 현대적인 제품이 아니라, 현행 기능을 안정적으로 이식하고 조직이 실제로 운영 가능한 선택지가 우선이다.

2. 기술 검토 항목

영역 검토 항목 고려사항
FE현행 화면 동등성복잡한 화면 흐름, 접근성, 브라우저 지원, 대체 UI 구현 난이도
BE로직 이식성프레임워크 변경 시 트랜잭션, 배치, 예외 처리 재현 가능성
DBDB 마이그레이션 도구DDL 변환, 데이터 이관, 정합성 검증 자동화 지원 여부
Infra클라우드 이전현행 네트워크, 보안 규정, 운영 조직 숙련도와의 적합성
공통인증/권한 체계기존 계정 통합, SSO 연계, 감사 이력, 권한 세분화 가능성
Infra관제 도구로그/메트릭 수집 범위, 알람 정책, 운영자 사용성
Infra배포 자동화현행 형상 전략과 승인 프로세스를 수용할 수 있는지 검토
공통보안/라이선스오픈소스 정책, 상용 라이선스 비용, 보안 취약점 대응 체계

3. 기술 전환 난이도 평가

영역 현행→목표 전환 난이도(1-5) 핵심 리스크
FEJSP/템플릿 → SPA/컴포넌트4화면 동등성 및 사용자 학습 비용
BE모놀리식 → 모듈형 서비스4숨은 업무 규칙 누락과 API 변경 영향
DB상용 DBMS A → DBMS B5프로시저/함수 이식성과 데이터 정합성
Infra온프레미스 → 클라우드3보안 심의와 네트워크 정책 재구성
공통수동 배포 → CI/CD3운영 승인 절차와 자동화 충돌
공통부분 관제 → 통합 모니터링2로그 표준화와 알람 튜닝 초기 비용

4. 전환 도구 목록

  • DB: ora2pg, AWS SCT, DMS, Flyway, Liquibase
  • Code: OpenRewrite, SonarQube, Understand, AI Code Assistant
  • Infra: Terraform, Ansible, Helm, Argo CD, Jenkins
  • Test: Postman/Newman, JMeter, Playwright, DB 비교 스크립트
  • Monitoring: Prometheus, Grafana, ELK/OpenSearch, APM 도구
