FE/BE 전환 설계

FE 화면 전환과 BE API/로직 전환을 병행 추진하기 위한 전략, 매핑, 역호환 방안을 정리한 설계 문서입니다.

1. FE 전환 전략 비교

전략 설명 장점 단점 적용 시기
전면 Rebuild 기존 화면을 신규 프레임워크로 일괄 재구축 일관된 UX/코드 품질 확보 초기 비용과 전환 리스크가 큼 화면 수가 제한적이고 구조 혁신이 필요한 경우
Strangler Fig 기존 화면을 유지하면서 기능 단위로 신규 화면으로 점진 전환 단계별 배포와 리스크 분산 가능 공존 구간의 복잡도 증가 대규모 레거시를 운영 중단 없이 전환할 때
Micro Frontend 도메인별 독립 배포 가능한 FE 모듈로 분리 전환 조직/도메인 단위 병렬 개발 용이 빌드/라우팅/디자인 통제가 어려움 여러 팀이 병행 개발하고 도메인 경계가 명확할 때

2. BE 전환 전략 비교

전략 설명 장점 단점 적용 시기
리팩토링 기존 언어/프레임워크를 유지한 채 구조를 개선하며 전환 학습 비용이 낮고 기존 자산 재사용 가능 근본적 한계가 남을 수 있음 기술 부채는 크지만 플랫폼 유지가 가능한 경우
언어/프레임워크 전환 핵심 서비스를 신규 언어/프레임워크로 재구현 성능, 유지보수성, 채용 경쟁력 개선 재학습과 회귀 테스트 비용이 큼 현 플랫폼 단종, 생산성 저하, 운영 한계가 명확할 때
API 우선 전환 도메인 API부터 신규 구조로 분리하고 내부 로직을 단계적으로 치환 대외 인터페이스 기준으로 전환 통제 가능 내부 복잡성이 일정 기간 유지됨 외부 연계가 많고 계약 기반 전환이 필요한 경우

3. FE 전환 상세

화면 전환 매핑 예시

AS-IS 화면 TO-BE 화면 전환 방식 우선순위
로그인 JSPReact 로그인 페이지신규 개발 후 라우팅 전환상
주문 조회 화면주문 통합 대시보드Strangler Fig상
고객 상세 팝업고객 상세 모달 컴포넌트컴포넌트 단위 치환중
운영자 통계 화면BI 연계 분석 화면기능 재설계 후 전환중

컴포넌트 전환 계획

  • 공통 UI(버튼, 입력폼, 모달, 테이블)부터 디자인 시스템 기반으로 표준화
  • 업무 화면은 라우트 단위로 우선순위를 부여하고 재사용 가능한 컴포넌트로 분해
  • 상태관리는 화면 복잡도에 따라 전역/도메인 스토어로 분리
  • 기존 브라우저 호환 요구사항과 접근성 기준을 신규 FE 설계에 반영

4. BE 전환 상세

API 전환 매핑 예시

AS-IS API TO-BE API 변환 내용 영향도
/legacy/login.do/api/v1/auth/login세션 방식 → 토큰 기반 인증상
/order/list.do/api/v1/orders조회 조건 표준화, 페이징 규격 통일상
/customer/detail.do/api/v1/customers/{id}식별자 경로화, 응답 스키마 정규화중
/stats/monthly.do/api/v1/reports/monthly배치 결과 조회 API로 재구성중

비즈니스 로직 이식 전략

  • 그대로 이식: 규칙이 안정적이고 외부 계약 영향이 큰 핵심 로직
  • 수정: 중복 제거, 예외 처리 보강, 트랜잭션 경계 조정이 필요한 로직
  • 재작성: 레거시 종속성이 심하거나 성능/확장성 개선이 필수인 로직

5. API 역호환성 전략

대외 연동과 기존 FE의 안정적인 병행 운영을 위해 일정 기간 역호환 계층을 둡니다. 구버전 엔드포인트는 신규 서비스로 포워딩하되, 요청/응답 변환 어댑터를 두어 계약 차이를 흡수합니다.

  • 구버전 URL 유지 + 신규 API로 내부 라우팅
  • 응답 필드명, 날짜 포맷, 코드값 변환 어댑터 제공
  • 버전 헤더 또는 경로(`/v1`, `/v2`) 기반 점진 전환
  • Deprecated API 사용량 모니터링 후 단계적 종료

6. AI 프롬프트

AI 프롬프트 — FE/BE 전환 설계
당신은 레거시 애플리케이션 컨버팅 프로젝트의 FE/BE 전환 설계자입니다.

입력 정보:
- AS-IS 기술 스택과 화면/서비스 목록
- TO-BE 기술 스택과 목표 아키텍처
- 외부 연동 시스템, 인증 방식, 배포 제약
- 우선 전환 대상 업무와 운영 중단 허용 범위

다음 항목이 포함된 "FE/BE 전환 설계서"를 작성하세요.
1. FE 전환 전략 비교와 선정 근거
2. BE 전환 전략 비교와 선정 근거
3. 화면 전환 매핑표와 우선순위
4. 공통 컴포넌트/디자인 시스템 전환 계획
5. API 전환 매핑표와 영향도 분석
6. 비즈니스 로직 이식 전략(그대로 이식/수정/재작성)
7. API 역호환성 및 병행 운영 방안
8. 리스크, 테스트 포인트, 단계별 추진 순서

실무 산출물 형식으로 표와 체크리스트를 적극 활용해 작성하세요.
입력에 화면 목록과 API 목록을 넣으면 전환 매핑표를 훨씬 구체적으로 생성할 수 있습니다.
